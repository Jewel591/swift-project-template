## 本地数据缓存方案

### 说明

当数据源为远端服务（Supabase、ASC API 等）时，冷启动需要等待网络请求完成才能渲染 UI。本地缓存策略可实现「打开即可见」的体验。

可选方案对比：

| 方案 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| JSON 文件缓存 | 只读展示数据（统计、时间线、列表） | 零依赖、实现简单、Codable 直接序列化 | 不支持复杂查询 |
| SwiftData 本地镜像 | 需要离线编辑、复杂查询的数据 | SwiftUI 原生集成、支持 @Query | 需维护本地模型与 DTO 映射、手动处理同步冲突 |
| UserDefaults | 极少量键值数据 | 最简单 | 不适合结构化数据和大体量存储 |
| NSCache / 内存缓存 | 临时数据（API 响应、计算结果） | 自动内存管理 | 不持久化，重启后丢失 |

常用策略：**cache-then-network** — 冷启动先读本地缓存渲染 UI → 后台静默拉取最新数据 → 有变化时更新 UI 并回写缓存。
